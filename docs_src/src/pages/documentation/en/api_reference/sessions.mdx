Once the Gotham Police Department could authenticate, Batman wanted each officer's dashboard to remember small bits of state between requests ‚Äî the theme they picked, a CSRF token, the last case they viewed ‚Äî without standing up a database for it. So he reached for Robyn's sessions.

## Sessions

Robyn provides signed-cookie sessions. The session is a small dictionary that is stored on the client inside a tamper-proof cookie (an HMAC-SHA256 signature over the data). No server-side store is required, and the signature means a client cannot modify the contents without invalidating it.

<b>
  Note: the session is signed, not encrypted. The data is encoded, so the client can read it ‚Äî never store secrets (passwords, raw tokens) in the session.
</b>

Enable sessions once on your app with `configure_sessions`, then read or write the session inside any handler through `request.session` ‚Äî exactly like `request.identity` for authentication.

## Configuration

`configure_sessions` accepts the following keyword arguments to control the cookie:

- `secret_key` (required) ‚Äî key used to sign the cookie.
- `cookie_name` ‚Äî name of the session cookie. Defaults to `"session"`.
- `max_age` ‚Äî session lifetime in seconds, also used as the cookie `Max-Age`. Defaults to 14 days. Pass `None` for a cookie that lasts only for the browser session.
- `path` ‚Äî cookie `Path`. Defaults to `"/"`.
- `domain` ‚Äî cookie `Domain`. Defaults to `None`.
- `secure` ‚Äî only send the cookie over HTTPS. Defaults to `False`; set it to `True` in production.
- `http_only` ‚Äî hide the cookie from JavaScript. Defaults to `True`.
- `same_site` ‚Äî `"Strict"`, `"Lax"`, or `"None"`. Defaults to `"Lax"`.

## How it works

Under the hood, `configure_sessions` registers a global `before_request` middleware that loads and verifies the cookie into a `Session` and attaches it to `request.session`, and a global `after_request` middleware that signs and writes the session back to the response when it has been modified. Because `request.session` is the same object across the `before_request`, handler, and `after_request` phases, your in-handler changes are exactly what gets written back.

If you want explicit control instead of the automatic middleware, you can use the `SessionManager` directly with `manager.load(request)` and `manager.save(session, response)`.
